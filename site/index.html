Why orbit-core

Most monitoring stacks drift into brittle pipelines: metrics in one place, events in another, alerts that never reach anyone, and dashboards glued to ad-hoc queries. orbit-core is built to stay fast, predictable, and easy to operate.

Unify metrics + events

Store both telemetry types in one canonical core and query them through a stable API.

Alerts — webhook & Telegram

Configurable threshold and absence rules evaluated every 60 s. Fire on firing and auto-resolve on resolved.

Auto-correlation

Background worker links metric anomalies (z-score + relative change) to concurrent events automatically.

Docker-ready

Self-contained docker compose up -d — no Node.js needed on the host. Migrations run automatically on first boot.

Deterministic connectors

No-AI shippers you can run from cron 24/7 with state files, fingerprint deduplication and bounded batches.

Rollups + retention

Plain Postgres with automatic rollups. Defaults: raw 14d, 5 m rollup 90d, 1 h rollup 180d.

Bounded queries

Auto-bucket + Top-N group-by to avoid runaway query cost and cardinality explosions.

AI-assisted dashboards

Optional Claude-powered builder generates validated DashboardSpec from natural language. AI is assistive only, never in automated pipelines.

What it is

  • An API-first storage + query core for metrics and events.
  • A connector pattern: deterministic ingestion with deduplication and auditability.
  • An alert engine: configurable rules evaluated every 60 s, dispatching to channels.
  • Dashboards as a validated spec (portable, versionable, reviewable).

What it is not

  • Not a full SIEM replacement.
  • Not "AIOps magic" running production pipelines.
  • Not unbounded SQL / infinite-cardinality by default.
  • Not a PagerDuty replacement (yet).

How it works

Connectors ship deterministic batches to ingest endpoints. The API validates, stores and serves queries. Three background workers handle rollups, anomaly correlation and alert evaluation continuously.

Core endpoints

  • GET /api/v1/health — build + DB status
  • GET /api/v1/metrics/prom — Prometheus exporter
  • POST /api/v1/ingest/metrics — MetricPoints
  • POST /api/v1/ingest/events — Events (fingerprint dedup)
  • POST /api/v1/query — OrbitQL time-series query
  • GET /api/v1/catalog/* — assets / metrics / dimensions
  • GET /api/v1/correlations — anomaly ↔ event links
  • * /api/v1/alerts/rules — CRUD alert rules
  • * /api/v1/alerts/channels — webhook + Telegram channels
  • GET /api/v1/alerts/history — notification log
  • * /api/v1/dashboards — CRUD dashboards (JSON spec)
  • POST /api/v1/ai/dashboard — AI-assisted generation

Storage strategy

metric_points raw ≤ 14d
metric_rollup_5m downsample ≤ 90d
metric_rollup_1h downsample ≤ 180d
Query engine selects the source table automatically based on the requested time range.

Background workers

rollup 5 m + 1 h aggregation + retention
correlate anomaly detection (z-score)
alerts rule evaluation every 60 s
See docs/ARCHITECTURE.md for full details.

What's New

Recently shipped

Features shipped in the latest release cycle.

🔔 Alerts

Webhook & Telegram notifications

Configurable alert rules (threshold + absence conditions). Evaluated every 60 s. Notifications dispatched on firing and resolved transitions via generic webhook or Telegram Bot API. Silence, enable/disable and full history included.

🐳 Docker

Self-contained Docker setup

Multi-stage Dockerfile + docker-compose.yml. Migrations run automatically on first boot via a one-shot migrate service. No Node.js or pnpm required on the host — deploy from a fresh machine in minutes.

🔗 Correlations

Anomaly auto-correlation

Background worker detects metric spikes (z-score + relative change) and links them to concurrent events. Results surfaced in the Correlations tab and via GET /api/v1/correlations.

🧹 Deduplication

Event fingerprint dedup

Ingest uses ON CONFLICT (fingerprint) DO UPDATE — connectors can re-ship the same event safely without creating duplicates. Unique partial index enforces integrity at the database level.

⚡ Reliability

Rate limiting & connection pooling

300 req/min per API key or IP with IPv6 support. Postgres pool with idle, connection and statement timeouts. X-Request-ID on every request for log correlation.

📊 Dashboards

AI-assisted dashboard builder

Claude-powered assistant generates DashboardSpec from natural language. Specs are validated server-side before saving — AI is assistive, the schema is the contract.

Quickstart

Up and running in under 5 minutes. Requires Docker ≥ 24 and Docker Compose ≥ 2.20.

Docker (recommended)
# 1. Clone
git clone https://github.com/rmfaria/orbit-core
cd orbit-core

# 2. Configure
cp .env.example .env
# Set a random API key:
#   sed -i '' "s/ORBIT_API_KEY=/ORBIT_API_KEY=$(openssl rand -hex 32)/" .env

# 3. Build images (~3 min on first run)
docker compose build

# 4. Start stack  (postgres → migrations → api → ui)
docker compose up -d

# 5. Verify
curl http://localhost/orbit-core/api/v1/health
# → {"ok":true,"db":"ok","workers":["rollup","correlate","alerts"]}

# Open the UI:  http://localhost/orbit-core/

What just started

  • postgres — PostgreSQL 16 with a persistent named volume
  • migrate — applies all 11 migrations, then exits (0)
  • api — Express API with 3 background workers
  • ui — nginx serving the React SPA + proxying /orbit-core/api/

Configure your first alert

# Create a Telegram channel
curl -X POST localhost/orbit-core/api/v1/alerts/channels \
  -H "Content-Type: application/json" \
  -d '{"id":"tg","name":"Ops","kind":"telegram",
  "config":{"bot_token":"TOKEN","chat_id":"CHATID"}}'

# Create a threshold rule (cpu > 85 avg 5 min)
curl -X POST localhost/orbit-core/api/v1/alerts/rules \
  -H "Content-Type: application/json" \
  -d '{"name":"CPU high","namespace":"nagios",
  "metric":"cpu","channels":["tg"],
  "condition":{"kind":"threshold","op":">",
    "value":85,"window_min":5,"agg":"avg"}}'

Development mode

Need hot-reload? Run API and UI natively (requires Node.js ≥ 22 and pnpm):

pnpm install
export DATABASE_URL='postgres://postgres:postgres@localhost:5432/orbit'
pnpm api:dev   # port 3000
pnpm ui:dev    # port 5173

Production hardening

  • Set ORBIT_API_KEY to a strong random secret.
  • Change POSTGRES_PASSWORD from the default.
  • Put a TLS reverse proxy (nginx / Caddy) in front on port 443.
  • See INSTALL.md for the full guide including Docker Swarm.

Connectors

Connectors are deterministic, cron-friendly and no-AI. They run on monitored servers — not inside the orbit-core stack.

Primary

Nagios (Python)

Ships perfdata as metrics and HARD state changes as events. Global event handler + cron pollers with byte-offset state.

connectors/nagios/INSTALL.md

Ready

Wazuh (Python)

Ships Wazuh alerts from alerts.json or OpenSearch API. Rule level → severity mapping + timestamp cursor.

connectors/wazuh/INSTALL.md

Ready

n8n (Python + workflow)

Ships execution errors and stuck runs. Polling mode via REST API + optional realtime Error Trigger workflow.

connectors/n8n/INSTALL.md

Via Wazuh

Fortigate (syslog → Wazuh)

Fortigate sends syslogs (UDP/TCP 514) to Wazuh. Events arrive as namespace=wazuh, kind=fortigate and are surfaced separately in the UI.

connectors/fortigate/INSTALL.md

Connector contract: any system that can do a POST /api/v1/ingest/metrics or POST /api/v1/ingest/events with a JSON payload and an X-Api-Key header is a connector. No SDK required.

Security

  • No secrets in the browser — API key lives in localStorage, never baked into built assets.
  • Prefer TLS + ORBIT_API_KEY in front of the API in any network-exposed deployment.
  • Rate limiting (300 req/min per key/IP, IPv6-safe) on all non-health endpoints.
  • Zod schema validation on all ingest and alert payloads — malformed input is rejected at the boundary.
Security policy: SECURITY.md
